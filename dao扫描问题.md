问题总结

1. 具体错误信息为：
   No qualifying bean of type 'cn.choicelink.selenium.modules.communal.dao.LinkAccessRecordsMapper' available
   这个错误表明 Spring 容器在启动时无法找到 LinkAccessRecordsMapper 这个 Bean，导致依赖注入失败。尽管 LinkAccessRecordsMapper 接口位于主类所在包的子包内，但由于 MyBatis 的特殊机制，Spring 并不会自动扫描到该接口。

2. 主要原因
   - MyBatis Mapper 扫描问题：
     虽然 Spring Boot 默认会扫描主类所在包及其所有子包下的组件（如 @Component, @Service, @Repository 等），但并不会自动扫描 @Mapper 注解标记的接口。
     即使 LinkAccessRecordsMapper 在正确的子包下，并且加上了 @Mapper 和 @Repository 注解，没有额外配置的情况下，Spring 容器依然无法识别它作为 Bean。
   - Mapper 包路径未被正确扫描：
     LinkAccessRecordsMapper 所在的包路径为 cn.choicelink.selenium.modules.communal.dao，而主类所在的包是 cn.choicelink.selenium，虽然它们之间存在父子关系，但由于 MyBatis 的特殊性，需要显式地告知 Spring 哪些包应该被扫描以发现 Mapper 接口。
   
3. 解决方案
     使用 @MapperScan 注解：
     在主类上添加 @MapperScan 注解来指定包含 Mapper 接口的包路径。这是最直接有效的解决办法。
     
     
     
4. 示例代码如下：

     ```java
     @SpringBootApplication
     @ServletComponentScan
     @MapperScan("cn.choicelink.selenium.modules.*.dao") // 使用通配符匹配多个模块下的 dao 包
     public class SeleniumApplication {
         // 主类内容...
     }
     
     /*
     如果你希望更精确地控制扫描范围，可以使用具体的包名或多个包名列表。
     每个 Mapper 加 @Mapper 注解（不推荐）：
     对于每个 Mapper 接口都添加 @Mapper 注解，这样 MyBatis 可以单独识别这些接口并注册为 Bean。但是这种方法维护成本较高，尤其是在有大量 Mapper 接口时。
     ```
